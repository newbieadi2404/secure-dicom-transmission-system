# Secure DICOM Management and Transmission System Prototype

## Abstract
This project presents a prototype for the secure handling and distribution of medical imaging data in the DICOM (Digital Imaging and Communications in Medicine) format. The system addresses the challenges of data confidentiality and integrity during transmission between healthcare providers and patients. By implementing a hybrid cryptographic pipeline, the system ensures that sensitive pixel data is protected via AES-256-GCM, while session keys are securely exchanged using RSA-2048. A role-based access control (RBAC) model is enforced through a Flask-based REST API, providing distinct interfaces for doctors, patients, and administrators. The prototype demonstrates a zero-knowledge-oriented data model where the server does not possess user private keys, and decryption occurs exclusively at the recipient's endpoint. Engineering challenges such as managing the memory overhead of large DICOM pixel arrays and the latency trade-offs inherent in hybrid encryption wrapping were central to the development process. Key metrics such as entropy and NPCR are monitored via an administrative dashboard to verify cryptographic strength. While the current implementation uses SQLite and local key storage, it serves as a technical foundation for scalable, cloud-based medical data exchange systems.

## Problem Statement
Standard medical image transmission often lacks end-to-end encryption, exposing sensitive patient data to unauthorized access during transit. Furthermore, ensuring the authenticity of medical records (verifying the sender) and protecting against data tampering or replay attacks are critical requirements that are often inadequately addressed in legacy healthcare IT systems. A unified system is needed that provides confidentiality, integrity, and authenticity for DICOM records in a distributed environment.

## Objectives
- **Hybrid Encryption Pipeline**: Implement a secure file distribution mechanism using AES-256-GCM and RSA-2048.
- **Role-Based Access Control (RBAC)**: Develop distinct permission models for Doctors, Patients, and Admins.
- **Digital Signatures**: Integrate RSA-PSS signatures to ensure data authenticity and non-repudiation.
- **System Integrity Monitoring**: Provide administrative tools for audit logging and cryptographic metrics (Entropy, NPCR, UACI).
- **Client-Side Decryption**: Enable local decryption of medical images to maintain a zero-knowledge data model.

## System Overview
The workflow supports multi-directional secure communication:
1. **Doctor-to-Patient**: Doctors upload DICOM records and transmit them to patient accounts via the hybrid encryption tunnel.
2. **Doctor-to-Doctor**: Healthcare providers can securely share imaging records with colleagues for consultation or referral.
3. **Patient-to-Doctor**: Patients can upload their own imaging data (e.g., from external sources) and securely transmit it to a registered doctor's inbox.
4. **Administrative Oversight**: Admins monitor audit trails and security metrics (Entropy, NPCR, UACI) to ensure system health and cryptographic compliance.

## System Architecture
The system implements a three-tier architecture with the following technology stack:
- **Frontend (React 19 + Vite)**: Chosen for its robust state management capabilities, which are essential for handling the lifecycle of large DICOM pixel data and ensuring component reusability across complex medical viewing interfaces.
- **Backend (Flask + SQLAlchemy)**: Selected for its seamless integration with specialized Python libraries such as `pydicom` and `cryptography`. Its lightweight nature allows for high-performance REST API prototyping without the overhead of more complex frameworks.
- **Database (SQLite)**: Used for lightweight, serverless metadata storage during the prototype phase.
- **Security Layers**: JWT (JSON Web Tokens) for session-based authentication and application-layer hybrid encryption.

## Security Design
- **Hybrid Encryption (AES + RSA)**: AES-256 in GCM (Galois/Counter Mode) is used for bulk data encryption. GCM was specifically selected because it provides both confidentiality and data authenticity (AEAD), preventing unauthorized bit-flipping or tampering during transit. RSA-2048 (OAEP) is utilized exclusively for secure session key encapsulation.
- **Digital Signatures**: RSA-PSS with SHA-256 is used to sign packages. This ensures that the record originates from a verified doctor and provides non-repudiation, as the signature can only be generated by the holder of the doctor's private key.
- **RBAC**: Enforced at the API layer via JWT claims. Middleware-level verification ensures that users only access resources permitted by their cryptographically signed role (Patient, Doctor, Admin).
- **Replay Protection**: Implemented via a nonce-based tracking system. A unique nonce is generated using a cryptographically secure pseudo-random number generator (CSPRNG) for each package, stored in the database, and validated during processing to ensure that a validly signed record cannot be re-processed maliciously.

## Implementation Details
Key modules of the prototype include:
- **Authentication**: Utilizes `Flask-JWT-Extended` to manage stateless sessions. JWTs are signed using HMAC-SHA256 and verified by backend decorators to enforce role-based access.
- **Encryption Pipeline**: A modular Python service that handles the sequential generation of a 256-bit AES session key, GCM encryption of the payload, RSA wrapping of the session key, and final RSA-PSS signing of the entire package.
- **DICOM Processing**: Uses `pydicom` to parse the DCM file structure and `numpy` to extract raw pixel data. The data is normalized into arrays before being serialized for the encryption pipeline.
- **Audit Logging**: A centralized service captures request metadata, user identity, and operation outcomes, providing an immutable trail of all security-critical events (e.g., failed decryption attempts).

## Performance Consideration
The prototype typically handles DICOM files ranging from 5MB to 50MB. The hybrid encryption process introduces a measurable latency overhead (typically <200ms for standard files) primarily due to the asymmetric wrapping of session keys and digital signature generation. The primary performance trade-off is the CPU-intensive nature of the cryptographic operations versus the reduced network bandwidth required for the compressed encrypted packages.

## Challenges Faced
- **Large File Management**: Handling large DICOM pixel arrays in memory during the encryption and serialization process required careful resource management to avoid memory exhaustion in the Flask environment.
- **Key Distribution Complexity**: Implementing a Certificate Authority (CA) model for public key distribution while maintaining a zero-knowledge-oriented data model presented significant synchronization challenges.
- **Cryptography Integration**: Merging low-level cryptographic primitives (AES-GCM, RSA-PSS) with high-level application logic required strict adherence to data padding and serialization standards to ensure cross-platform compatibility.
- **Frontend Data Handling**: Processing and visualizing large `numpy`-derived pixel data in the browser required optimized React rendering to maintain UI responsiveness.

## Limitations
- **SQLite Usage**: Not suitable for high-concurrency production environments; lacks the robust features of a relational DBMS like PostgreSQL or MySQL.
- **Key Management**: RSA private keys are currently stored as local PEM files, which is a significant security risk compared to using a dedicated Hardware Security Module (HSM) or Cloud KMS.
- **Email-Based Transmission**: Standard SMTP is used for distributing packages, which may be restricted by attachment size limits and lacks the inherent security of dedicated medical exchange protocols.

## Future Enhancements
- **Database Migration**: Transition from SQLite to PostgreSQL for better scalability, concurrency, and data integrity.
- **Secure Key Management**: Integrate AWS KMS, Azure Key Vault, or HashiCorp Vault for secure, centralized key lifecycle management.
- **Network Security**: Enforce HTTPS/HSTS and implement mutual TLS (mTLS) for all client-server communication.
- **Scalable Deployment**: Containerize services using Docker and orchestrate with Kubernetes for high availability and load balancing.

## Conclusion
The prototype successfully demonstrates a secure DICOM transmission system using modern cryptographic standards. By integrating hybrid encryption and digital signatures, the system achieves a high degree of data confidentiality and authenticity. The development process highlighted the importance of balancing cryptographic overhead with system performance and the necessity of robust audit mechanisms in healthcare IT environments.
